Release Notes for X-CUBE-WIFI1

Software for STM32, Expansion for STM32Cube

Copyright 2016 STMicroelectronics

 


The X-CUBE-WIFI1 software package is an expansion for STM32Cube. This software provides software APIs running on STM32 for STM's X-NUCLEO-IDW01M1 or X-NUCLEO-IDW04A1. It is built on top of STM32Cube software technology that ease portability across different STM32 microcontrollers.

X-CUBE-WIFI1 software features:
  • Complete middleware to build applications using the Wi-Fi module (SPWF01Sx or SPWF04Sx).
  • Easy to use abstract APIs to configure and use SPWF01Sx or SPWF04Sx.
  • Easy portability across different MCU families thanks to STM32Cube.
  • Sample applications that the developer can use to start experimenting with the code.
  • Free user-friendly license terms.
  • Examples implementation available for X-NUCLEO-IDW01M1 STM32 expansion board plugged on top of one NUCLEO-F103RB, NUCLEO-F401RE, NUCLEO-L476RG or NUCLEO-L053R8.
  • Examples implementation available for X-NUCLEO-IDW04A1 STM32 expansion board plugged on top of one NUCLEO-F401RE or NUCLEO-L476RG only.

The figure below shows the overall architecture.
     
         

  • At the bottom layer there are the HW components: the STM32 MCU and the X-NUCLEO-IDW01M1 or X-NUCLEO-IDW04A1 Wi-Fi expansion board.

  • The drivers abstract low-level details of the hardware and allow the middleware components and applications to access the SPWF04Sx or the SPWF01Sx device in a hardware independent fashion.
  • The applications provide examples of how to use the APIs.

Known Limitations

  • For the X-NUCLEO-IDW04A1, higher level application protocols are not supported like MQTT, Websocket, TFTP, etc. File APIs, ping API, server socket close for all clients and standby/wakeup APIs are not supported. The APIs seen for websocket and TFTP are under development and testing and not fully functional
  • For this current first release on X-NUCLEO-IDW04A1, please limit each receive data chunk to 740 bytes for sockets and 512 bytes for HTTP
  • On X-NUCLEO-IDw04A1 VCOM application at higher baud-rates (above 115200) may be problematic with missing characters during reception.
  • Network scan using UART interface on X-NUCLEO-IDW04A1 will yield a maximum of 10 results for each call
  • Turning off the radio (AT+S.WIFI=0) during the initialization phase might result in stalling, with no response from the module. Please reset the module/nucleo in this case.
  • Sometimes the FW might stop receiving SPI headers after prolonged data reception
  • NUCLEO-F103RB does not work correctly with X-NUCLEO-IDW01M1 when UART baud-rate 921600 is selected.
  • NUCLEO-F103RB must be run at it's highest frequency of 72 MHz
  • It may occur that the module is not able to connect to certain Access Points and the module will repeatedly keep scanning in a loop. This behaviour is known.
  • On Nucleo-L053R8, only anonymous and one-way authentication is possible to be executed. On NUCLEO-L053R8, mutual authentication currently cannot be run due to memory size limitation.
  • Remote configuration is currently not supported
  • A maximum of 50 events can be pending at any given time and any more events after this will be dropped (not queued)
  • Warning
    • SPI peripheral can go in timeout for F4 MCU families, due to a bug in the HAL_SPI_TransmitReceive() function. In case of problems, you can call the __HAL_SPI_CLEAR_OVRFLAG() macro to clear the overrun flag before using the HAL_SPI_TransmitReceive() function.
